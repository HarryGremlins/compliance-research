> **目标**：分析 Kaleido 和 Canton Network 如何满足机构隐私合规需求，识别各自的优势和不足，找出 Altius L1 的差异化发力点。
>
> **聚焦领域**：fintech、银行、机构金融、支付
>
> **日期**：2026-03-17

## 2. 隐私架构对比

### 2.1 Canton Network：协议层隐私

Canton 的隐私与大多数区块链根本不同。它**不依赖加密或混淆** — 而是基于**协议层的数据最小化**。

#### Daml 隐私模型的两个核心原则

1. **原则一**：参与方只能看到自己有利益关系（stake）的操作。
2. **原则二**：看到某个操作的参与方，同时看到该操作的所有（传递性的）后果。

"有利益关系"的定义：
- 是该操作的必要授权方
- 是被操作合约的签名方（signatory）
- 是合约的观察方（observer），且该操作创建或归档了该合约

#### 子交易隐私（Sub-transaction Privacy）

这是 Canton 最核心的特性：

> "子交易隐私意味着交易参与者只能了解到他们直接参与的那部分交易，而不知道交易的其他任何部分。这既适用于交易内容，也适用于其他参与者的身份。"

**实际例子**：如果 Alice、Bob、欧元银行、美元银行共同参与一笔贸易结算，欧元银行只看到欧元那条腿，美元银行只看到美元那条腿，两家银行互不知道对方的存在。这不是靠加密实现的，而是从根本上就不传输无关数据。

#### 私有合约存储（Private Contract Store）

- 每个参与者节点管理自己的**私有合约存储**
- **没有全局状态**，**没有全局合约存储**
- Canton 协议保证所有本地存储同步形成一个"虚拟全局账本"，同时保持隐私
- 每个参与者的 offset（相当于区块高度）是唯一的，不同节点之间不可比较

#### Need-to-Know 数据分发

> "即使是加密形式的（子）交易，也只会传输给有知情权的参与者节点；同时，参与者节点会被告知所有他们有权知道的（子）交易。"

> "隐私不是因为合约数据被加密为密文来保障的；而是因为账本模型确保没有经过验证的知情权的参与者，根本不会以任何形式收到该交易。"

#### 同步器架构

- **排序器（Sequencer）**：转发加密的定向消息，保证顺序。不知道消息内容。
- **调解器（Mediator）**：协调交易确认，不看到完整交易细节。
- **同步域身份管理器**：管理身份注册，只转发身份更新，不能凭空创造。
- 参与者节点之间通过同步域进行**端到端加密**通信。

#### 泄露问题（Divulgence）

Canton 承认一个隐私上的微妙问题："泄露"（divulgence）— 当某些合约作为参与方有利益关系的操作的传递性后果被创建时，参与方可能看到本不属于自己的合约。在设计隐私敏感型应用时需要格外注意。

### 2.2 Kaleido：多层隐私工具箱

Kaleido 的方法本质不同 — 它提供**一系列可选的隐私工具**，覆盖多种协议，而不是单一的协议层隐私模型。

#### 私有交易管理器

- **旧方案**：Tessera (Quorum) — 链下点对点加密通信给指定参与方。非参与方只看到哈希证明。
- **新方案**：Tessera 已从 Hyperledger Besu 移除。Kaleido 现在推荐 **Paladin**（Linux Foundation Decentralized Trust 项目）用于 EVM 隐私。

#### Paladin：EVM 可编程隐私

Paladin 是 Kaleido 目前推荐的隐私方案，提供三个隐私域：

| 隐私域 | 技术 | 用途 |
|---|---|---|
| **Zeto** | 零知识证明（ZKP） | 基于 ZKP 的代币转账，隐藏金额和参与方 |
| **Noto** | 公证人验证 | 中心化验证实现高效私有交易 |
| **Pente** | 私有 EVM 执行 | 隔离的智能合约执行环境 |

关键特点：
- 兼容**任何未修改的 EVM 链** — 无需改造底层账本
- UTXO 模型支持并行处理和高效私有状态管理
- 跨域原子结算，统一私有和公有账本状态
- 选择性披露给授权方

#### 零知识代币（Anonymous Zether）

Kaleido 内置基于 Anonymous Zether 协议（JPMorgan/Stanford/Visa Research）的 ZKP 代币：
- 将 ERC20 代币充值 → 获得屏蔽的 ZTH 代币
- ZTH 转账时隐藏金额和参与方身份（使用诱饵账户）
- 随时可提现回 ERC20
- 使用 Σ-Bullets（Bulletproofs + Sigma 协议）— 无需可信设置

局限性：每个账户每个 epoch 只能进行一笔转出；匿名集大小是性能与隐私的权衡；目前仅限 Quorum 环境。

#### 链下方案

- **应用间消息传递**：Apache Kafka 骨干网，端到端 PKCS#7 加密载荷，链上地址簿
- **链下文档存储**：文件经哈希、签名、压缩和非对称加密。链上仅存哈希引用。支持 AWS S3 和 Azure Blob。
- **HD 钱包**：分层确定性钱包生成一次性签名密钥 — 隐藏交易方身份

#### 其他隐私手段

- **可信执行环境（TEE）**：文档中有介绍但非原生集成
- **多方计算（MPC）**：提及适用于研究/分析场景
- **Hyperledger Fabric 通道**：基于通道的数据隔离

### 2.3 核心对比

| 隐私维度 | Canton Network | Kaleido |
|---|---|---|
| **隐私层级** | 协议原生（Daml 账本模型） | 应用层（工具箱方式） |
| **数据最小化** | 内置：参与方永远不会收到无关数据 | 取决于所选工具；部分方案仍广播证明 |
| **子交易隐私** | 是 — 核心特性 | 无原生等价物；可通过私有通道近似实现 |
| **ZKP 支持** | 协议层不内置 | 有 — Anonymous Zether、Paladin Zeto |
| **身份隐私** | 非相关方看不到参与者身份 | HD 钱包隐藏身份；ZKP 中使用诱饵账户 |
| **智能合约隐私** | Daml：基于利益相关方的可见性设计 | Pente：隔离的私有 EVM 执行 |
| **链下数据** | 明确建议链上/链下分离 | 文档存储 + 消息传递，带加密 |
| **EVM 兼容** | ❌ 不兼容（仅 Daml） | ✅ 原生支持 |

## 3. 合规与安全

### 3.1 认证与审计

| 认证 | Canton Network | Kaleido |
|---|---|---|
| **SOC 2 Type II** | 未公开声明 | ✅ A-lign 出具（安全性、可用性、保密性） |
| **ISO 27001** | 未公开声明 | ✅ ISMS 认证 |
| **ISO 27017** | 未公开声明 | ✅ 云安全控制 |
| **ISO 27018** | 未公开声明 | ✅ 公有云 PII 保护 |
| **ISO 27701** | 未公开声明 | 未公开声明 |
| **PCI DSS** | 未公开声明 | 通过 AWS/Azure 基础设施间接满足 |

**关键发现**：两个平台都没有公开声明 ISO 27701（隐私信息管理系统）认证 — 而这是与机构隐私合规最直接相关的标准。

### 3.2 安全基础设施

**Canton Network**：
- 端到端加密的参与者通信
- 通过同步域身份管理器实现密码学身份管理
- 多方确认阈值机制用于交易授权
- 无全局状态暴露 — 通过数据最小化实现安全

**Kaleido**：
- AES-256 加密所有静态文件系统和密钥材料
- TLS 1.2 用于所有传输中数据
- 云 HSM 集成（AWS KMS、Azure Key Vault）用于交易签名
- VPC Private Link 支持私有网络流量
- 盐值哈希凭证验证（不存储明文）
- 每个区块链网络的防火墙隔离环境
- 供应链安全（OpenSSF 签名的软件包）
- 年度正式风险评估

### 3.3 身份与 KYC

| 能力 | Canton Network | Kaleido |
|---|---|---|
| **身份模型** | Daml 参与方 = 密码学密钥标识符；托管在验证者节点上 | 组织身份注册表，基于 X.509 PKI 证书 |
| **KYC 集成** | 不内置；委托给应用层 | 通过 Oracle 集成（Chainlink）+ 私有文档存储实现链下 KYC |
| **身份注册** | 同步域身份管理器（去中心化） | 链上注册服务（可插拔，已被 EEA 规范 v5 采纳） |

## 5. 对标 ISO 27701 隐私控制措施

### 5.1 区块链最难满足的 5 项控制

| ISO 27701 控制 | 要求 | Canton | Kaleido |
|---|---|---|---|
| **A.1.3.7 — 删除权** | 用户可访问、更正或删除其 PII | ⚠️ 无全局状态，数据仅存在相关参与者节点上，节点层面可删除。但 divulgence 可能扩散数据。 | ⚠️ 链下文档存储支持删除。链上数据不可变。推荐 PII 链下 + 哈希上链模式。 |
| **A.1.3.11 — 自动化决策** | 纯自动化处理做出影响用户决定时须说明 | ⚠️ Daml 合约确定性且透明（可读），但无内置通知机制 | ⚠️ 标准智能合约透明性；无内置 GDPR 第 22 条通知机制 |
| **A.1.4.6 — 处理结束后删除/去标识化** | 处理结束后删除 PII 或使其无法识别 | ✅ 私有合约存储按参与者隔离；归档合约是 Daml 的原生操作。无"永久全局存储" | ⚠️ 链上数据不可变；须使用密码学粉碎或链下存储模式 |
| **A.1.5.2 — 跨境传输依据** | 跨境传输 PII 须有法律依据 | ⚠️ 多同步器架构允许地理分离，但跨同步器交易可能跨境 | ⚠️ 支持多区域部署并可选择云区域（美/欧/亚太），但跨区域环境是可能的 |
| **A.1.2.8 — 联合控制者** | 多方共同治理数据处理的协议 | ✅ Daml 的签名方/观察方模型显式定义各方角色。Canton Foundation 治理架构 | ✅ 联盟治理 + 可继承权限模型 + 多会员制 |

### 5.2 更广泛的控制覆盖

| 控制领域 | Canton | Kaleido |
|---|---|---|
| **A.1.4.2 — 采集限制** | ✅ 数据最小化是协议特性 | ⚠️ 可通过链下模式实现，但非协议强制 |
| **A.1.4.3 — 处理限制** | ✅ 参与方只处理自己相关的子交易 | ⚠️ 私有交易限制可见性，但不限制处理范围 |
| **A.1.4.5 — PII 最小化** | ✅ 核心设计原则 — 子交易隐私 | ⚠️ ZKP 代币可隐藏金额/身份；通用 PII 最小化是应用层任务 |
| **A.1.4.10 — PII 传输控制** | ✅ 端到端加密；need-to-know 分发 | ✅ TLS 1.2 + PKCS#7 消息加密；私有交易 |
| **A.3.25 — 日志记录** | ✅ 按参与方审计追踪 + OpenTelemetry | ✅ 区块浏览器 + 日志流 |
| **A.3.26 — 密码学控制** | ✅ 协议级加密；基于身份系统的密钥管理 | ✅ AES-256 静态加密、TLS 传输加密、HSM 集成 |
| **A.3.29 — 安全架构** | ✅ 协议层的 Privacy-by-Design | ⚠️ 安全的基础设施，但隐私是应用层关注点 |

## 6. Gap 分析 — 两个平台都未完全覆盖的领域

### 6.1 共同缺失

| 缺口 | 描述 | Altius 机会 |
|---|---|---|
| **ISO 27701 认证** | 两者都没有。这是机构买家最直接的 PIMS 合规证明。 | Altius 可以成为首个获得 ISO 27701 的区块链平台。 |
| **原生同意管理** | 两者都不提供原生的同意追踪（A.1.2.4-5），委托给应用层。 | 在协议/SDK 层面内置同意管理框架。 |
| **自动化决策透明** | 两者都没有内置机制满足 GDPR 第 22 条（自动化决策通知）。而智能合约天然就是自动化的。 | 内置自动化决策审计钩子 + 可解释性。 |
| **数据主体权利门户** | 两者都没有提供开箱即用的方案让 PII 主体行使访问/更正/删除权。 | 面向终端用户的自助隐私权利管理。 |
| **跨境合规自动化** | 两者都支持多区域部署，但都不自动执行数据驻留规则。 | 管辖区感知的数据路由：自动根据配置规则（如 PIPL、PDPA）执行数据本地化。 |
| **隐私影响评估工具** | 两者都不提供 PIA/DPIA 工具，假设由外部完成。 | 集成 DPIA 工作流，在智能合约部署/更新时自动触发。 |

### 6.2 Canton 特有的缺口

| 缺口 | 描述 |
|---|---|
| **协议层无 ZKP** | 数据最小化虽然强大，但 Canton 不提供 ZKP 用于"在不暴露数据的前提下证明某件事"（如合规证明、AML 检查）。 |
| **无正式合规认证** | 没有 SOC 2、没有 ISO 27001/27017/27018。依赖参与机构自身的合规体系。 |
| **Daml 锁定** | 智能合约必须用 Daml 编写 — 一种领域特定语言，不兼容 EVM。开发者生态有限。 |
| **Divulgence 泄露** | 泄露机制可能无意中将合约暴露给非预期的参与方，需要谨慎的应用设计。 |

### 6.3 Kaleido 特有的缺口

| 缺口 | 描述 |
|---|---|
| **无协议层隐私** | 隐私是应用层关注点，不是协议保证。开发者必须选择并正确实现隐私工具。 |
| **无子交易隐私** | 私有交易的所有参与者看到完整交易。无法在单笔交易内实现部分可见性。 |
| **中心化 BaaS 模型** | 作为托管平台，Kaleido 是一个中心化的控制点。创建信任依赖。 |
| **Paladin 成熟度** | Paladin（推荐的隐私方案）是相对较新的 LF 项目。远不如 Canton 多年验证的隐私模型成熟。 |
| **无原生数据最小化** | 数据最小化需由开发者自行实现 — 协议不强制。 |

## 7. Altius L1 的战略意义

### 7.1 结合会议上下文的分析

根据 3/16 会议纪要，Altius L1 的 Middleware Layer 包含四个模块：

1. **Compliance Standards**（ISO 20022 等合规标准支持）
2. **Opt-in Privacy**（FHE、ZK、TEE、Viewing Key）
3. **Transfer Hooks**（黑白名单、转账限额等）
4. **Native Stablecoin as Gas**（单/多 Gas Token）

基于此架构，以下是与竞品对比后的定位建议：

### 7.2 定位矩阵

```
                    协议层隐私
                       ↑
                       |
                Canton  |
                       |
  认证少/开发者少 ------+------→ 认证多/开发者多
                       |
                       | ← Altius 目标区域
                       |
                Kaleido |
                       ↓
                    应用层隐私
```

**Canton**：协议层隐私最强，但无合规认证，不兼容 EVM，Daml 开发者生态有限。
**Kaleido**：认证齐全、多协议支持，但隐私停留在应用层，无子交易隐私。
**Altius 机会**：EVM 兼容 + 内置合规（Middleware Layer）+ 可选隐私（ZK/TEE/Viewing Key）。

### 7.3 五个差异化方向

| # | 方向 | 说明 | 对应 Altius 架构 |
|---|---|---|---|
| 1 | **合规原生** | 把 ISO 20022 等合规要求直接编码进 Token Standard（如 Memo 字段）和 Transfer Hooks（黑白名单、限额检查） | Middleware: Compliance Standards + Transfer Hooks |
| 2 | **可选隐私 + 合规证明** | Canton 有数据最小化但无 ZKP；Kaleido 有 ZKP 但无协议层保障。Altius 用 Viewing Key 实现选择性披露，用 ZKP 实现合规证明（如"此地址未在黑名单上"而不暴露名单） | Middleware: Opt-in Privacy |
| 3 | **管辖区感知** | 一国一链的模型天然支持数据本地化。不同国家不同链、不同规则、不同稳定币 — 根据合规 research 的结果配置每条链的参数 | 整体架构（SDK 模式） |
| 4 | **合规仪表盘** | Canton 和 Kaleido 都没有面向企业管理者的合规配置和审计界面。Altius 规划的 Enterprise Layer Dashboard 正好填补这个空白 | Enterprise Layer: Dashboard |
| 5 | **EVM 兼容 + Rust 原生性能** | Canton 牺牲了 EVM 兼容性换取隐私；Kaleido 虽然支持 EVM 但性能受限于标准实现。Altius 用 Rust 原生实现 Token 逻辑（非合约），保持 EVM 兼容的同时获得性能优势 | Infrastructure Layer |

### 7.4 合规 Research 对 Altius 产品的直接影响

Tony 在会议中明确了一个关键观点：

> "我们要去来研究那些合规的研究，就是每家银行它都是要符合同样的规则的，我们把这些所有的规则都实现了，它就只需要上来填参数了。"

这意味着合规 research 的最终产出不仅是一份报告，更是**可编码进 Transfer Hooks 和 Compliance Standards 的具体规则集**。例如：

- **ISO 20022**：转账必须支持 Memo 字段 → Token Standard 必须包含此字段
- **FATF Travel Rule**：转账超过一定额度必须附带发送方/接收方身份信息 → Transfer Hook 检查
- **各国制裁名单**：地址黑名单检查 → Transfer Hook 前置检查
- **GDPR 删除权**：链上不存 PII → 架构设计原则，PII 链下 + 哈希上链
- **Viewing Key + 合规审计**：监管方通过 Viewing Key 查看特定交易 → 选择性披露

### 7.5 建议的认证优先级

| 优先级 | 认证 | 理由 |
|---|---|---|
| 🔴 P0 | **ISO 27701** | 两个竞品都没有。与隐私合规最直接相关。 |
| 🔴 P0 | **SOC 2 Type II** | 企业准入门槛。Kaleido 有，Canton 没有。 |
| 🟡 P1 | **ISO 27001** | 信息安全管理基础。Kaleido 有。 |
| 🟡 P1 | **ISO 27017/27018** | 云安全 + 云 PII 保护。Kaleido 有。 |
| 🟢 P2 | **PCI DSS** | 如果涉及支付卡数据流则需要。 |

## 8. 总结

| 维度 | Canton Network | Kaleido | Altius 机会 |
|---|---|---|---|
| **隐私模型** | 协议原生，最强数据最小化 | 应用层工具箱 | 结合两者：内置合规规则 + 可选 ZK/TEE 隐私 |

| **未被满足的需求** | 合规自动化、ZKP 证明 | 子交易隐私、数据最小化 | 合规原生区块链 |
