# ISO/IEC 27701:2025 深度解读

> **文档来源**：ISO/IEC 27701:2025(en) 第二版，2025年10月发布
> **全称**：信息安全、网络安全与隐私保护 — 隐私信息管理系统 — 要求与指南
> **适用读者**：金融/区块链行业的初学者，希望理解机构级隐私合规要求
> **与我们研究的关系**：这是全球公认的隐私管理"操作手册"，Kaleido 等企业级平台的 ISO 认证体系核心组成部分

---

## 一、这份文档到底是什么？（给初学者的白话解释）

### 1.1 一句话概括

ISO 27701 是一份**教组织如何系统性地保护个人隐私信息的国际标准**。你可以把它理解为一本"隐私保护操作手册"——它不告诉你具体的法律条文（那是 GDPR、CCPA 的事），而是告诉你**应该建立什么样的管理体系和流程**来确保你能遵守任何国家的隐私法。

### 1.2 为什么这份标准很重要？

打个比方：
- **GDPR**（欧盟通用数据保护条例）告诉你"你必须保护用户隐私，否则罚款"
- **ISO 27701** 告诉你"这是保护用户隐私的具体步骤和方法清单"

对于金融机构来说，获得 ISO 27701 认证就像拿到了一张"隐私保护合格证"——它证明你有一套完整的体系来管理个人数据，而不是随意处理。

### 1.3 2025 版的重大变化

这是第二版（取代了 2019 年第一版）。最关键的变化是：

> **2025 版被重新起草为独立的管理体系标准。**

这意味着什么？2019 版只是 ISO 27001（信息安全管理）的"扩展包"——你必须先有 ISO 27001 才能用它。而 2025 版**可以独立使用**，说明国际标准组织认为隐私管理已经重要到需要自己单独成为一个完整的标准。

## 二、核心概念（你需要先理解的"术语表"）

在读这份标准之前，你需要理解几个关键概念。这些概念贯穿全文：

### 2.1 PII（Personally Identifiable Information，个人可识别信息）

**定义**：任何可以直接或间接识别到某个具体自然人的信息。

**举例**：
- 直接识别：姓名、身份证号、手机号、邮箱地址
- 间接识别：IP 地址、设备指纹、cookie ID、**区块链钱包地址**（这一点很关键！）

**在区块链/Crypto 语境下**：以太坊地址（0x...）本身是伪匿名的，但如果能通过 KYC 记录、交易所绑定等方式关联到具体个人，它就变成了 PII。这也是为什么链上数据的隐私问题如此复杂。

### 2.2 PII 控制者（PII Controller）

**定义**：决定"为什么要处理个人数据"和"怎么处理个人数据"的组织。

**白话**：谁说了算，谁就是控制者。

**举例**：
- 一家加密交易所收集用户的 KYC 信息（姓名、护照号）→ 交易所是 PII 控制者
- 一家 DeFi 协议的治理 DAO 决定收集用户邮箱用于空投通知 → DAO 是 PII 控制者

### 2.3 PII 处理者（PII Processor）

**定义**：受 PII 控制者委托，代为处理个人数据的组织。

**白话**：帮别人干活的。

**举例**：
- 交易所使用第三方 KYC 服务商（如 Jumio、Sumsub）来验证用户身份 → KYC 服务商是 PII 处理者
- 一家企业使用 Kaleido 的 BaaS 平台部署区块链节点 → Kaleido 是 PII 处理者
- 一家银行使用 Canton Network 进行代币化证券结算 → Canton Network 是 PII 处理者

### 2.4 PII 主体（PII Principal）

**定义**：PII 所对应的那个自然人——就是"数据主人"。

**举例**：你在交易所开户，你就是 PII 主体。你的姓名、证件号、交易记录都是你的 PII。

### 2.5 PIMS（Privacy Information Management System，隐私信息管理系统）

**定义**：一套专门用来管理隐私保护的组织体系，包括政策、流程、角色分配、技术措施等。

**白话**：就是一家公司专门用来"管好用户隐私"的整套制度和工具。

## 四、正文核心要求详解（第 4-10 章）

### 4.1 了解组织背景（第 4 章）

**标准要求组织首先搞清楚几件事**：

1. **你是谁？** — 确定自己是 PII 控制者还是 PII 处理者（或两者都是）
2. **你面临什么环境？** — 适用的隐私法律、法规、司法裁决、合同要求
3. **谁是利益相关方？** — 包括 PII 主体本人、监管机构、客户、业务伙伴
4. **你的 PIMS 范围是什么？** — 必须包含 PII 的处理

**区块链场景下的特殊挑战**：

在传统企业中，"你是谁"很好回答。但在区块链/Web3 场景中，这个问题变得模糊：
- 一个 DAO 是 PII 控制者吗？（它没有传统的法律实体）
- 一个智能合约是 PII 处理者吗？（它是自动执行的代码）
- 区块链节点运营商算什么角色？（他们存储了全网数据但并不"使用"它）

标准特别指出：**如果一个组织同时担任控制者和处理者，必须分别确定角色，每个角色适用不同的控制措施**（第 4.1 章）。

### 4.2 领导力与隐私政策（第 5 章）

**核心要求**：高层管理者必须：
- 确保隐私政策与组织战略方向一致
- 将 PIMS 要求融入业务流程
- 提供必要资源
- 传达隐私管理的重要性

**隐私政策必须**：
- 形成书面文档
- 在组织内部传达
- 向相关方公开

**对金融机构的意义**：这不是 IT 部门的事，而是**董事会级别**的事。如果一家想服务金融机构的区块链平台（如 Kaleido、Canton Network）没有高层挂帅的隐私政策，机构客户不会信任它。

### 4.3 规划：隐私风险评估与处置（第 6 章）

这是标准中**最核心**的章节之一，包含三个关键流程：

#### 隐私风险评估（6.1.2）

组织必须建立一个流程来：
1. 设定风险接受标准
2. 识别隐私风险（对组织和对 PII 主体的风险都要考虑）
3. 分析风险的后果和可能性
4. 确定风险等级
5. 与风险标准比较，排出优先级

#### 隐私风险处置（6.1.3）

确定如何处理识别出的风险：
1. 选择合适的风险处置方案
2. 确定所有必要的控制措施
3. 将自己确定的控制措施与**附录 A 的清单**对比，确保没有遗漏
4. 编制**适用性声明**（Statement of Applicability）— 列出所有必要的控制措施及其纳入/排除的理由
5. 制定风险处置计划

**信息安全项目至少应涵盖**：
- 信息安全风险管理
- 信息安全政策
- 访问控制
- 网络安全管理
- 开发安全
- 供应商管理
- 事件管理
- **密码学**（cryptography）— 这一点对区块链特别相关！
- 物理和环境安全

### 4.4 运营、监控与改进（第 8-10 章）

这些章节要求：
- **定期执行**隐私风险评估（而不是一次性的）
- **定期内部审计** PIMS 是否有效运行
- **高层管理评审** — 定期审视整个体系的适宜性
- **不符合项处理** — 发现问题时如何纠正、防止复发

## 五、附录 A：隐私控制措施完整清单（最实用的部分）

附录 A 是这份标准中**最具操作价值**的部分——它给出了所有你需要考虑的隐私控制措施的完整清单。

### 5.1 PII 控制者的控制措施（Table A.1）

分为 5 大类：

#### 第一类：采集与处理条件

| 编号 | 控制措施 | 白话解释 |
|------|----------|----------|
| A.1.2.2 | 识别并记录目的 | 你收集用户数据要干嘛？必须说清楚、写下来 |
| A.1.2.3 | 识别合法依据 | 你凭什么收集？必须有法律依据（同意/合同/合法利益等） |
| A.1.2.4 | 确定何时、如何获取同意 | 什么时候问用户要同意？用什么方式？ |
| A.1.2.5 | 获取并记录同意 | 必须有记录证明用户同意了 |
| A.1.2.6 | 隐私影响评估 | 新处理或变更处理前，评估对隐私的影响 |
| A.1.2.7 | 与 PII 处理者的合同 | 你外包给别人处理数据？必须签合同约束他们 |
| A.1.2.8 | 联合 PII 控制者 | 多方共同决定数据处理？明确各方责任 |
| A.1.2.9 | PII 处理记录 | 维护你处理了哪些数据的记录 |

**Crypto/Web3 语境**：这一类措施意味着，任何收集用户 KYC 信息的项目（交易所、合规 DeFi 协议、RWA 平台）都必须清楚记录：为什么收集、法律依据是什么、用户是否同意、外包给谁处理、处理了什么。

#### 第二类：对 PII 主体的义务

| 编号 | 控制措施 | 白话解释 |
|------|----------|----------|
| A.1.3.2 | 确定对 PII 主体的义务 | 弄清楚法律要求你对用户承担什么义务 |
| A.1.3.3 | 确定提供给 PII 主体的信息 | 你必须告诉用户哪些信息？什么时候告诉？ |
| A.1.3.4 | 向 PII 主体提供信息 | 用清晰、易懂的方式告知用户 |
| A.1.3.5 | 提供修改/撤回同意的机制 | 用户必须能修改或撤回他们的同意 |
| A.1.3.6 | 提供反对处理的机制 | 用户必须能反对你处理他们的数据 |
| A.1.3.7 | 访问、更正或删除 | 用户有权访问、更正或**删除**他们的数据 |
| A.1.3.8 | 通知第三方义务 | 用户修改/撤回/反对后，你必须通知已共享数据的第三方 |
| A.1.3.9 | 提供已处理 PII 的副本 | 用户要求时，提供他们数据的副本 |
| A.1.3.10 | 处理请求 | 建立处理用户请求的政策和流程 |
| A.1.3.11 | 自动化决策 | 如果用纯自动化处理做出影响用户的决定，必须说明 |

**Crypto/Web3 语境**：A.1.3.7（删除权）是区块链面临的**最大挑战**——链上数据不可变，如何"删除"？A.1.3.11（自动化决策）与智能合约直接相关——如果一个智能合约自动决定拒绝某个用户（比如基于 AML 评分），这就涉及自动化决策的义务。

#### 第三类：隐私设计与默认隐私

| 编号 | 控制措施 | 白话解释 |
|------|----------|----------|
| A.1.4.2 | 限制采集 | 只收集必要的最少数据 |
| A.1.4.3 | 限制处理 | 只做必要的最少处理 |
| A.1.4.4 | 准确性与质量 | 确保数据准确、完整、最新 |
| A.1.4.5 | PII 最小化目标 | 定义数据最小化目标，使用去标识化等技术 |
| A.1.4.6 | 处理结束后的去标识化和删除 | 不再需要时，删除或使数据无法识别个人 |
| A.1.4.7 | 临时文件 | 处理过程中产生的临时文件必须定期清理 |
| A.1.4.8 | 保留期限 | 不得保留超过必要期限 |
| A.1.4.9 | 处置 | 必须有数据销毁的政策和流程 |
| A.1.4.10 | PII 传输控制 | 传输 PII 时必须有适当的安全措施 |

**Crypto/Web3 语境**：A.1.4.5（PII 最小化）和 A.1.4.6（去标识化）是区块链隐私技术最直接的应用场景。零知识证明（ZKP）就是实现数据最小化的完美工具——你可以证明某人满足某个条件（如年龄>18），而不暴露具体年龄。

#### 第四类：PII 的共享、转移与披露

| 编号 | 控制措施 | 白话解释 |
|------|----------|----------|
| A.1.5.2 | 识别跨境传输依据 | 跨国传输数据的法律依据是什么？ |
| A.1.5.3 | 可传输的国家和组织 | 列出可以传输 PII 的国家和国际组织 |
| A.1.5.4 | PII 传输记录 | 记录所有 PII 的传入和传出 |
| A.1.5.5 | PII 披露给第三方的记录 | 记录向谁、什么时候披露了什么 PII |

**Crypto/Web3 语境**：区块链天然是跨境的——节点分布在全球各地。如果一个许可链（如 Canton Network）的节点分布在新加坡、伦敦和纽约，那每笔交易中的 PII 都在跨境传输。A.1.5.2 和 A.1.5.3 要求你必须有跨境传输的法律依据，并记录可以传输到哪些国家。

### 5.2 PII 处理者的控制措施（Table A.2）

PII 处理者（如 Kaleido、云服务商）有一套对应但不同的控制措施：

**关键差异**：
- 处理者必须**按客户指令**处理数据，不能自行决定
- 处理者**不得**将数据用于营销/广告（除非获得 PII 主体的明确同意）
- 处理者如果认为客户的指令违法，**必须通知客户**
- 处理者必须**披露是否使用了分包商**，并允许客户反对分包商变更

### 5.3 控制者和处理者共同的安全控制（Table A.3）

这部分有 31 项安全控制措施，与我们研究最相关的包括：

| 编号 | 控制措施 | 与 Crypto/区块链的关系 |
|------|----------|----------------------|
| A.3.5 | 信息分类 | 需要将 PII 纳入信息分类体系 |
| A.3.7 | 信息传输 | 链上/链下数据传输都需要安全措施 |
| A.3.8 | 身份管理 | 区块链身份（DID）管理的合规基础 |
| A.3.9 | 访问权限 | 谁可以访问 PII？许可链的权限管理 |
| A.3.14 | 记录保护 | PII 相关记录的完整性保护——区块链天然优势！ |
| A.3.25 | 日志记录 | 记录谁、何时、访问了什么 PII |
| A.3.26 | **密码学使用** | 必须定义和实施加密规则，包括密钥管理 |
| A.3.27 | 安全开发生命周期 | 智能合约开发必须包含隐私保护 |
| A.3.29 | 安全系统架构 | 系统架构必须遵循"隐私设计"和"默认隐私"原则 |
| A.3.31 | 测试信息 | **不得用真实 PII 做测试！** 应使用虚假或合成数据 |

## 六、附录 B：实施指南精华（最有实操价值的部分）

附录 B 对附录 A 中的每项控制措施给出了详细的实施指南。以下提取与我们研究最相关的内容：

### 6.1 合法依据（B.1.2.3）

PII 处理的合法依据可以包括：
- PII 主体的**同意**
- 履行**合同**需要
- 遵守**法律义务**
- 保护 PII 主体的**切身利益**
- 执行**公共利益**任务
- 控制者的**合法利益**

**特殊类别 PII**（如健康信息、种族、宗教、生物识别数据、基因数据）受到更严格的控制。

**Crypto 语境**：AML/KYC 场景下，交易所收集用户身份信息的合法依据通常是"遵守法律义务"（反洗钱法）。但如果同一份 KYC 数据被用于营销推荐，那就需要另一个合法依据（通常是同意）。

### 6.2 隐私影响评估（B.1.2.6）

**什么时候需要做隐私影响评估？**
- 涉及**自动化决策**且产生法律效果
- 大规模处理**特殊类别 PII**
- 对公共场所的**系统性监控**

评估需要确定的要素：
- 处理的 PII 类型清单
- PII 存储在哪里
- PII 可能被传输到哪里
- 数据流图和数据地图

### 6.3 联合 PII 控制者协议（B.1.2.8）

当多方共同决定数据处理时，协议应包含：
- 共享/联合控制的目的
- 各 PII 控制者的身份
- 共享或传输的 PII 类别
- 处理操作概述
- 各方的角色和责任
- 技术和组织安全措施的实施责任
- **PII 泄露时的责任划分**（谁通知、什么时候通知）
- PII 的保留和处置条款
- 不履行协议的责任

**Crypto 语境**：这直接适用于**联盟链**场景。例如 Canton Network 有多个金融机构参与，每个参与方可能都是联合 PII 控制者——他们需要签署这样的协议。

### 6.4 删除权与去标识化（B.1.3.7, B.1.4.6）

标准要求：
- 组织应实施政策和机制，使 PII 主体能够**访问、更正或删除**其 PII
- 处理结束后，应**删除 PII 或使其无法识别个人**
- 去标识化技术包括：**泛化**（如将精确年龄变为年龄段）、**随机化**（如添加噪声）

**Crypto 语境**：这是区块链隐私合规的**核心矛盾**。链上数据不可删除，但标准要求可以删除。解决方案：
1. **链下存储 PII + 链上存储哈希**：只在链下存储个人数据，链上仅存不可逆的哈希值。删除链下数据即可满足要求。
2. **Crypto-shredding**（密码学粉碎）：用加密密钥加密链上的 PII，需要"删除"时销毁密钥。数据仍在链上，但没有密钥就无法解读。
3. **零知识证明**：根本不在链上存储 PII，只存储 ZK 证明。

### 6.5 密码学使用（B.3.26）

标准要求：
- 必须定义和实施**密码学使用规则**，包括**密钥管理**
- 某些司法管辖区**可能要求**使用密码学保护特定类型的 PII（如健康数据、身份证号、护照号、驾照号）
- 必须告知客户在什么情况下使用密码学保护 PII

**Crypto 语境**：区块链本身就是密码学的产物。这个控制措施要求不仅要*使用*密码学，还要*规范*密码学的使用——包括密钥管理（谁持有密钥？密钥如何轮换？密钥丢失怎么办？）。对于多签钱包、MPC 密钥管理等场景尤为重要。

### 6.6 安全事件响应（B.3.12）

**PII 泄露时必须做的事**：

对于 PII 控制者：
- 确定是否发生了需要响应的 PII 泄露
- 通知相关**监管机构**（某些司法管辖区有强制要求）
- 通知受影响的 **PII 主体**
- 保留完整记录，包括：事件描述、时间段、后果、报告人、处理步骤、影响范围

对于 PII 处理者：
- PII 泄露通知条款应写入与客户的合同
- 应在发现泄露后**立即**通知 PII 控制者
- 通知义务不延伸到由客户或 PII 主体造成的泄露

**Crypto 语境**：加密交易所被黑客攻击导致用户数据泄露时，必须按照以上流程通知用户和监管机构。2024-2025 年多起交易所安全事件中，是否及时通知用户已经成为监管处罚的重要考量。

## 七、附录 D：与 GDPR 的映射（极其有价值）

附录 D 提供了 ISO 27701 各条款与 GDPR 第 5-49 条的**逐条映射**。这意味着：

> **如果你按照 ISO 27701 建立了 PIMS，你基本上就满足了 GDPR 的大部分要求。**

几个关键映射示例：

| ISO 27701 条款 | GDPR 对应条款 | 说明 |
|---|---|---|
| B.1.2.3（合法依据） | GDPR 第 6 条（处理的合法性） | 6 种合法依据完全对应 |
| B.1.3.7（访问/更正/删除） | GDPR 第 16-17 条（更正权/删除权） | 直接对应被遗忘权 |
| B.1.2.6（隐私影响评估） | GDPR 第 35 条（DPIA） | 隐私影响评估的具体情形和方法 |
| B.1.5.2（跨境传输） | GDPR 第 44-49 条（向第三国传输） | 跨境传输的完整框架 |
| B.1.3.11（自动化决策） | GDPR 第 22 条（自动化决策） | 智能合约/AI 决策的合规依据 |

## 八、关于 Crypto/Web3 的内容：本文档的盲区

### 8.1 直接提及的内容

**ISO 27701 全文中没有直接提及区块链、加密货币、Web3、DeFi、智能合约或分布式账本技术。**

这并不意外——ISO 27701 是一份**通用的**隐私管理标准，设计为适用于所有行业和技术。它故意不针对特定技术，以保持广泛适用性。

### 8.2 间接相关的内容

虽然没有直接提及，但以下条款与 Crypto/Web3 高度相关：

| 标准中的表述 | Crypto/Web3 的对应场景 |
|---|---|
| "密码学的使用"（A.3.26） | 区块链的核心技术基础 |
| "去标识化技术"（A.1.4.5, A.1.4.6） | ZKP、同态加密、MPC |
| "自动化决策"（A.1.3.11） | 智能合约自动执行 |
| "PII 传输控制"（A.1.4.10） | 链上交易中的 PII 传输 |
| "联合 PII 控制者"（A.1.2.8） | 联盟链多方治理 |
| "分包商管理"（A.2.5.7-A.2.5.9） | 节点运营商、Oracle 提供者 |
| "跨境传输"（A.1.5.2-A.1.5.5） | 跨境区块链节点数据复制 |
| "安全系统架构原则"（A.3.29） | 智能合约和 DApp 的隐私设计 |
| "测试信息"（A.3.31） | 测试网上不得使用真实用户数据 |
| "日志记录"（A.3.25） | 区块链本身就是不可变的日志 |
| "处理结束后删除"（A.1.4.6） | 链上数据不可变性的核心矛盾 |

### 8.3 标准缺失但行业需要的内容

以下是 ISO 27701 **没有覆盖**但区块链行业迫切需要解决的问题：

1. **去中心化场景下的"数据控制者"认定**：当没有单一实体控制数据处理时，谁是控制者？
2. **链上数据的不可变性与删除权的冲突**：标准只说"应删除"，但没有给出链上场景的指导
3. **伪匿名地址是否构成 PII**：标准没有明确说区块链地址是否算 PII
4. **去中心化治理与 PIMS 管理责任**：DAO 如何建立 PIMS？谁是"高层管理者"？
5. **跨链数据流动的隐私管控**：资产/数据跨链桥接时的 PII 保护

## 九、对 Tabula 研究的启示

### 9.1 这份标准对我们意味着什么？

1. **ISO 27701 是机构客户的"入场券"**：金融机构在选择区块链平台合作伙伴时，ISO 27701 认证（或至少遵循其框架）很可能是基本要求。Kaleido 已经有 ISO 27018（云端 PII 保护），但尚未公开确认 ISO 27701 认证。

2. **附录 A 的控制清单是我们的"合规检查表"**：我们可以用附录 A 的控制措施作为框架，逐一评估各个区块链平台（Kaleido、Canton Network、R3 等）的合规程度。

3. **附录 D 的 GDPR 映射是"一举两得"**：研究 ISO 27701 的合规性，基本上同时也覆盖了 GDPR 的要求。

### 9.2 建议的下一步行动

1. **建立合规矩阵**：用附录 A 的 70+ 项控制措施作为行，各区块链平台作为列，逐一评估
2. **重点关注 5 个最困难的条款**：
   - A.1.3.7（删除权）— 链上不可变性冲突
   - A.1.3.11（自动化决策）— 智能合约合规
   - A.1.4.6（处理结束后去标识化/删除）— 数据生命周期管理
   - A.1.5.2（跨境传输）— 全球节点部署
   - A.1.2.8（联合控制者）— 联盟链治理
3. **调研各平台的技术解决方案**：ZKP、私有通道、链下存储等如何映射到这些控制措施
